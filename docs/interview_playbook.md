# 面试学习手册（随项目迭代）

## 1. 项目一句话
- 我做的是“金融资讯信息整合问答系统”，核心是可追溯检索，不是投资建议系统。

## 2. 当前可演示链路
1. 数据采集：`collect_data.py`
2. 数据验证：`verify_data_quality.py`
3. 数据清洗去重：`clean_and_prepare_data.py`
4. 检索演示：`query_demo.py` + `BM25Retriever`

## 3. 每一步的技术细节（面试高频）

### 3.1 数据采集
- 设计：统一 `BaseCrawler` 抽象，所有数据源走 `fetch -> parse -> save`。
- 为什么：后续接入新源（研报、社媒）只需新增子类，降低耦合。
- 风险：第三方接口字段漂移、限流、数据缺失。
- 对策：字段兜底映射、日志记录、失败告警、定期回归测试。

### 3.2 数据验证
- 指标：`valid_rate`、字段覆盖率（title/content/time/url 等）。
- 为什么：RAG 前的“数据可用性门槛”，避免垃圾进垃圾出。
- 结果解读：如果 `sampled` 很小（如新闻仅 8 条），先补采样本再评估。

### 3.3 清洗与去重
- 设计：标准化文本、过滤空值、最小内容长度过滤、稳定 ID 去重。
- 稳定 ID：用 `sha1(title + time + url + source)`，替代 Python `hash()`。
- 为什么：`hash()` 跨进程不稳定，不利于增量更新与一致性。

### 3.4 检索（BM25）
- 实现：`rank_bm25 + jieba`。
- 为什么先做 BM25：不依赖 GPU/向量库，能快速验证“检索是否命中”。
- 局限：语义泛化弱，后续需要向量召回 + 重排。

## 4. 高频追问与回答模板

### Q1: 为什么不是直接用大模型回答？
- A: 金融场景对时效性和可追溯要求高。纯生成有幻觉风险，RAG 能把证据链显式引入回答。

### Q2: 为什么先 BM25，不直接上向量库？
- A: 先用 BM25 建“最小可验证闭环”，确认数据质量和召回逻辑正确，再引入向量和重排优化效果。

### Q3: 你怎么衡量检索效果？
- A: 先用 Top-k 命中率和人工相关性打标；后续再做 MRR/nDCG 与线上问答反馈对齐。

### Q4: 你遇到过什么工程问题？
- A: 例如公告采集分支返回空、输出目录参数未贯通、依赖版本失效。我通过最小修复+验证脚本+依赖分层解决。

### Q5: 合规怎么保证？
- A: 系统定位信息整合；输出中加来源和风险声明；不输出买卖建议、收益承诺、价格预测。

## 5. 当前可讲的工程亮点
- 分阶段交付：先跑通采集和检索闭环，再上向量检索与 Agent。
- 可验证：每个阶段都有脚本和可量化指标。
- 可扩展：抽象层清晰，便于新增数据源和检索策略。

4. 再引入 Agent 工具路由（新闻/公告/计算）。
